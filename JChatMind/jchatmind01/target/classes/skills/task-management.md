# 多 Agent 任务拆解与调度系统 (Task Management System)

## 概述

本系统提供了一套完整的任务生命周期管理工具，允许你将复杂项目拆解为有序的小任务，并通过依赖关系（`blockedBy`）控制执行顺序。多个
Agent 可以协作：各自认领（claim）并完成（complete）分配给自己的任务，系统自动保证依赖顺序不被违反。

## 核心概念

### 任务生命周期

```
createTask()  →  [PENDING]  →  claimTask()  →  [IN_PROGRESS]  →  completeTask()  →  [COMPLETED]
                     ↑                                                                    │
                     └──────────────── 解除阻塞后续任务 ───────────────────────────────────┘
```

### 三种状态

| 状态            | 含义                 | 可执行操作                     |
|---------------|--------------------|---------------------------|
| `pending`     | 待处理，等待被认领          | `claimTask`（前提：所有前置任务已完成） |
| `in_progress` | 已被某 Agent 认领，正在执行中 | `completeTask`（只能由认领者完成）  |
| `completed`   | 已完成                | 无（会自动解除对依赖此任务的其他任务的阻塞）    |

### 依赖关系 (blockedBy)

- 创建任务时可以指定一个前置任务 ID 列表（`blockedBy`）
- 只有当所有前置任务都处于 `completed` 状态时，该任务才能被认领
- 如果 `blockedBy` 为空或未指定，任务可以立即被认领

## 可用工具

### 1. createTask — 创建任务

```
createTask(subject, description, blockedBy)
```

**参数**：

- `subject` (必填): 任务简短标题，如 "设计数据库表结构"
- `description` (必填): 任务详细描述，包含具体要求和验收标准
- `blockedBy` (可选): 前置任务 ID 列表，如 `["TASK-xxx", "TASK-yyy"]`

**使用场景**：当你需要将一个大项目拆解为子任务时，为每个子任务调用一次。

### 2. listTasks — 列出所有任务（摘要）

```
listTasks()
```

**返回**：所有任务的概览，按状态分组（待处理 / 进行中 / 已完成），**不包含详细描述**以节省 Token。

**使用场景**：

- 查看当前整体进度
- 发现可以认领的 `pending` 任务
- 检查哪些任务阻塞了后续任务

### 3. getTask — 获取任务详情

```
getTask(taskId)
```

**返回**：指定任务的完整信息，**包含详细描述**。

**使用场景**：认领任务前或执行任务时，需要了解任务的具体要求时使用。

### 4. claimTask — 认领任务

```
claimTask(taskId, agentName)
```

**约束**：

1. 任务必须存在
2. 所有前置任务必须已完成
3. 任务状态必须为 `pending`（不能被其他 Agent 重复认领）

**使用场景**：当你准备开始执行一个任务时，调用此工具将其标记为你的任务。

### 5. completeTask — 完成任务

```
completeTask(taskId, agentName)
```

**约束**：

1. 任务必须存在
2. 只能完成你自己认领的任务（`agentName` 必须与认领时一致）
3. 任务状态必须为 `in_progress`

**使用场景**：当你确认任务已完成，调用此工具标记完成。完成后系统会自动解除依赖此任务的其他任务的阻塞。

## 工作流程指南

### 标准流程：单 Agent 串行执行

当你独自处理一个复杂需求时：

1. **分析需求** → 确定需要拆分为几个步骤
2. **创建任务** → 为每个步骤调用 `createTask`，指定正确的依赖关系
3. **列出任务** → 调用 `listTasks` 确认任务结构正确
4. **逐个执行** → 对每个无依赖的待处理任务：
    - 调用 `claimTask` 认领
    - 执行具体工作（调用其他工具完成）
    - 调用 `completeTask` 标记完成
5. **检查进度** → 随时调用 `listTasks` 查看整体状态

### 高级流程：多 Agent 协作

当需要与其他 Agent 协作时：

1. **主 Agent 拆解**：主 Agent 调用 `createTask` 创建所有任务，建立依赖关系图
2. **子 Agent 认领**：每个子 Agent 调用 `listTasks` 查看待处理任务，调用 `claimTask` 认领适合自己的任务
3. **并行执行**：多个 Agent 同时工作，各自认领并完成自己的任务
4. **依赖自动流转**：当一个任务完成后，依赖它的任务自动解锁，可被其他 Agent 认领

## 最佳实践

### 任务拆解原则

1. **粒度适中**：每个任务应该是一个可独立完成的工作单元，不要太粗也不要太细
2. **描述清晰**：`description` 中应包含具体的验收标准和预期产出
3. **依赖最小化**：只标注真正的前置依赖，避免过度约束导致并行度降低
4. **命名规范**：`subject` 使用动词开头，如 "实现用户登录"、"编写单元测试"

### 依赖设计示例

```
项目：构建一个博客系统

TASK-A: "设计数据库表结构"           → 无依赖，可立即执行
TASK-B: "实现后端 API"              → 依赖 [TASK-A]
TASK-C: "实现前端页面"              → 依赖 [TASK-A]
TASK-D: "编写集成测试"              → 依赖 [TASK-B, TASK-C]

执行顺序：A 先完成 → B 和 C 可并行执行 → B 和 C 都完成后 D 才能开始
```

### 注意事项

- 认领任务前先用 `getTask` 查看详细描述，确认自己有能力完成
- 认领后尽快完成，避免长时间占用任务导致整体进度阻塞
- 如果认领后发现自己无法完成，目前系统不支持"放弃任务"，需要通知主 Agent 协调
- `listTasks` 返回的是摘要（不含描述），节省 Token；需要详情时再用 `getTask`
- 验证所有权：如果你发现 completeTask 报错“无权操作”，请先执行 getTask <taskId> 确认该任务当前状态及 owner 字段。如果
  owner 为空或不是你，请重新执行 claimTask。
- 依赖闭环：在尝试认领任务之前，务必通过 listTasks 确认该任务是否已被其他队友认领。
- 任务完成后务必同时 `completeTask` + `sendMessage(STATUS_UPDATE)`，让 Lead 感知进度
- 当无任务可做时，Teammate 应调用 `terminate`（系统会自动重启）
